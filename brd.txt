## Backend
- Runtime: Node.js with Express.js OR FastAPI (Python) — pick one, be consistent
- Language: TypeScript (Node) or Python (FastAPI)
- REST API with versioning: /api/v1/...
- WebSockets: Socket.io (Node) or websockets (Python) for real-time notifications

- Primary DB: PostgreSQL (via Prisma ORM for Node OR SQLAlchemy for Python)

- Search: PostgreSQL full-text search (baseline) + optional Elasticsearch/Meilisearch for advanced search
- Schema migrations: Prisma Migrate or Alembic

## AI/ML Services
- Image tagging: Google Cloud Vision API OR AWS Rekognition (auto-tagging: mountains, beach, sports, crowd, etc.)
- Facial recognition: AWS Rekognition Face Indexing API (collection-based face matching)
- AI captions: OpenAI GPT-4o Vision API (describe image → generate caption)
- Duplicate detection: perceptual hashing (pHash) via sharp (Node) or imagehash (Python)
- Image moderation: AWS Rekognition content moderation

## Cloud & Storage
- Object storage: AWS S3 (media files, watermarked downloads, thumbnails)
- CDN: AWS CloudFront (serve media via CDN with signed URLs)
- Image processing: Sharp (Node) or Pillow (Python) for resize, compress, watermark
- Watermark rendering: Canvas API (server-side) or Sharp composite
- Environment: AWS EC2 OR Railway/Render (for demo), with S3 mandatory

## Auth
- Strategy: JWT (access token short-lived 15min + refresh token 7d in httpOnly cookie)
- OAuth2: Google OAuth (NextAuth.js or Passport.js)
- Password hashing: bcrypt
- Role-based access: middleware guard on every protected route

## DevOps
- Containerization: Docker + docker-compose (frontend, backend, postgres, redis, worker)
- CI/CD: GitHub Actions (lint → test → build → deploy)
- Env management: .env.example with all required keys documented

  AI/ML PIPELINES — IMPLEMENT IN FULL

## Pipeline 1: Post-Upload Processing (async job via BullMQ/Celery)
When a file is uploaded to S3, enqueue a job that does:
1. Download file from S3 to temp
2. Compression: resize to max 2048px wide, compress JPEG quality 85 (Sharp/Pillow)
3. Generate thumbnail: 400x400 cover crop
4. pHash duplicate detection: compute perceptual hash, compare against existing hashes in DB (Hamming distance < 10 = duplicate warning)
5. AI tagging: call AWS Rekognition detect_labels → extract top 10 labels, store in media.ai_tags
6. Content moderation: call AWS Rekognition detect_moderation_labels → flag if unsafe content found (set moderation_status = pending if flagged)
7. AI caption generation: call GPT-4o Vision with the image → generate 1-sentence caption, store in media.ai_caption
8. Facial recognition: call AWS Rekognition index_faces → store all detected faces in face_index table
9. Face-to-user matching: for each detected face_id, query user_selfies table, call search_faces_by_image to find matching user → update face_index.user_id, create notification ("You were tagged in a photo at [event]")
10. Update media.upload_status = done
11. Emit WebSocket event to uploader: upload-complete with final media data

## Pipeline 2: Watermark on Download
When user requests download:
1. Check if watermarked version already exists in S3 (cache hit → serve immediately)
2. If not: download original from S3, overlay watermark using Sharp composite:
   - Club name (top-left)
   - Event name (bottom-left)
   - User role badge (bottom-right)
   - Subtle diagonal club logo watermark at 15% opacity
3. Upload watermarked version to S3 /watermarked/ prefix
4. Record download in downloads table
5. Return signed S3 URL (expires in 5 min) for browser download

## Pipeline 3: Facial Recognition — My Photos
When user uploads selfie (/profile/edit → selfie upload):
1. Upload selfie to S3
2. Call AWS Rekognition index_faces with the selfie → get face_id
3. Store in user_selfies (user_id, selfie_url, rekognition_face_id)
4. Run search_faces_by_image against the entire Rekognition collection
5. Find all media where matching faces were detected (face_index table)
6. Update face_index.user_id for all matches
7. Return list of matched media → show in /my-photos

## Pipeline 4: Smart Search
Build a search that queries across:
- media.ai_tags (array contains match)
- media.ai_caption (full-text)
- events.name, events.description (full-text)
- users.username (prefix match)
- albums.name
- Upload date range filter
Use PostgreSQL tsvector + GIN index for full-text. Return grouped results: events, media, users.

═══════════════════════════════════════════════
  CLOUD ARCHITECTURE — AWS SETUP
═══════════════════════════════════════════════

Implement the following (use AWS SDK v3):

S3 BUCKETS:
- {app}-media-original (private, IAM-only access)
- {app}-media-processed (private, CloudFront-served)
- {app}-watermarked (private, presigned URL download)
- {app}-avatars (public-read for profile pics)
- {app}-selfies (private, AI processing only)

CloudFront:
- Distribution for -media-processed bucket
- Signed URLs for private media (valid 1 hour)
- Public access for public media
- Cache policy: image/video files cached 7 days

S3 Upload flow:
- Backend generates presigned POST URL
- Browser uploads directly to S3 (bypasses backend, reduces server load)
- S3 triggers Lambda OR backend receives webhook → enqueues processing job

Image processing:
- All original images preserved in -original bucket
- Processed (compressed + thumbnail) stored in -processed
- CloudFront URL stored in media.cdn_url
